## 目的

ファイルの内容を OpenAI SDK で参照し、その結果に応じて分類先を振り分けるマイクロサービスを実装する。

想定する処理は次の通り。

1. ファイルのパス情報を受け取る
2. ファイル内容を抽出する
3. OpenAI SDK に内容を渡して分類ラベルを返す
4. 分類結果に応じて保存先や後続処理を切り替える

## 要件の整理

- 対象はテキストファイルだけでなく、将来的に PDF や Office 文書にも拡張したい
- 分類ラベルは固定のカテゴリ集合にしたい
- 推論結果は監査できるように保存したい
- 大きいファイルや失敗時にも、サービス全体が落ちないようにしたい

## ファイル種別の方針

最初からすべてのファイル種別を同じ扱いにしない。コストと抽出難易度が違うため、段階的に対応する。

### Phase 1: テキスト系を優先

- `txt`
- `md`
- `csv`
- `json`
- `log`

これらは抽出コストが低く、分類ロジックの検証に向いている。

### Phase 2: 文書系を追加

- `pdf`
- `docx`
- `xlsx`

これらはページ数やシート数で分割しやすいので、非同期処理との相性がよい。

### Phase 3: 画像・スキャン文書を追加

- `png`
- `jpg`
- `jpeg`
- `webp`

OCR が必要になるため、抽出段階を別モジュールに分けるのがよい。

### 非対応の扱い

- 想定外の MIME タイプは `unsupported_file` として弾く
- 必要なら人手確認キューへ回す

## OCR を低コスト・簡易にする方針

OCR はできるだけ最後の手段にする。まずは OCR しなくても読める経路を優先し、必要なときだけ限定的に実行する。

### 優先順位

1. 文字埋め込み済みの PDF やテキスト系ファイルは OCR しない
2. PDF はまずテキスト抽出を試す
3. 画像やスキャン文書だけ OCR に回す
4. OCR に失敗したページだけ再試行する

### 安くするコツ

- OCR 対象をページ単位に分ける
- すべてのページを一度に処理しない
- 解像度は必要最小限にする
- 文字領域が小さいページは後回しにする
- まず軽い前処理だけ行い、重い補正は避ける

### 実装候補

#### 候補1: ローカル OCR

- `tesseract` を直接使う
- もしくは `ocrmypdf` を使って PDF に文字層を付ける
- 外部 API を使わないのでコストが読みやすい

向いているケース:

- サーバ内で完結させたい
- 通信コストを増やしたくない
- 認識精度よりもコストと簡便性を優先したい

#### 候補2: 条件付き OCR

- まず通常抽出を試す
- 抽出文字数が極端に少ないページだけ OCR する
- 画像率が高いときだけ OCR する

向いているケース:

- OCR を常時回したくない
- できるだけ処理量を減らしたい

### 判定ルール例

- 1 ページあたりの抽出文字数が閾値未満なら OCR 対象
- 画像のみの PDF は OCR 対象
- テキスト抽出が十分に取れたページは OCR しない

### かなり簡易に始めるなら

- 対象はまず `pdf` のみ
- 文字抽出で十分なページはそのまま使う
- OCR は「抽出文字数が少ないページだけ」に限定する
- OCR エンジンはローカルの `tesseract` を使う

### 推奨結論

コストを最小化したいなら、**OCR は全ファイル対象にせず、テキスト抽出で失敗したページだけローカル OCR をかける** のが一番簡単で安い。

この方針なら、OpenAI SDK の利用対象も OCR 後のテキストだけに絞れるため、LLM のコストも抑えやすい。

## 分割仕様の方針

ファイルを丸ごと OpenAI に渡さず、抽出した本文をチャンクに分けて処理する。

### 分割の軸

1. ページ数
2. バイトサイズ
3. トークン数

### 推奨ルール

- 1 チャンクは 1,500〜3,000 トークン程度を上限にする
- 重複を避けつつ、文脈切れ対策で 100〜200 トークン程度のオーバーラップを入れる
- PDF はページ単位で切り、長いページだけさらに細かく分割する
- CSV は行単位で切る
- JSON はキー単位または配列要素単位で切る

### 分割後の扱い

- 各チャンクを個別に分類する
- 最後にチャンク結果を集約して最終カテゴリを決める
- 重要箇所が複数カテゴリにまたがる場合は、信頼度の高い上位候補を残す

## 実装上の推奨

### 抽出モジュール

- `extractText(file): string[]` のように、チャンク配列を返す形にする
- ファイル種別ごとに抽出器を切り替える

### 分類モジュール

- チャンクごとに OpenAI SDK を呼ぶ
- 返却値は必ず JSON に統一する
- `category`, `confidence`, `reason`, `chunkIndex` を保存する

### 集約モジュール

- 全チャンクの結果を集約して最終結果を決める
- 例: 最頻値、最大 confidence、ルールベース優先順位

## コスト抑制の考え方

- ファイル全体を毎回送らない
- チャンク上限を固定する
- 長文は要約してから最終分類する
- 同一ファイルの再分類結果はキャッシュする
- 失敗時の再試行回数に上限を設ける

## 実装案

### 案1: 同期APIで完結する最小構成

1. RoutingFile サービスがファイルを受け取る
2. 文字抽出を行う
3. OpenAI SDK で分類する
4. 結果をそのまま返す、または分類先を決めて保存する

#### 向いているケース

- まず PoC を作りたい
- リクエストごとに即時応答が必要
- ファイルサイズが小さい

#### 注意点

- 長時間かかると HTTP タイムアウトしやすい
- 大きなファイルや抽出失敗に弱い
- OpenAI の呼び出し失敗がそのまま UX に出る

## 案1の実装案 (Node.js + TypeScript + Express + Prisma)

案1では、RoutingFile サービスが HTTP リクエストを受けて、その場でファイル内容を参照し、OpenAI SDK で分類し、結果を Prisma で保存して返す。

### 想定ディレクトリ構成

```text
src/MicroService/RoutingFile/
├── package.json
├── tsconfig.json
├── prisma/
│   └── schema.prisma
└── src/
		├── index.ts
		├── db.ts
		├── routes/
		│   └── files.ts
		├── services/
		│   ├── fileReader.ts
		│   ├── classifier.ts
		│   └── routing.ts
		└── types/
				└── routing.ts
```

### 役割分担

#### `index.ts`

- Express 起動
- JSON パース
- ルーティング登録
- ヘルスチェック

#### `routes/files.ts`

- ファイル分類 API を公開する
- リクエストのバリデーションを行う
- `routing` サービスを呼ぶ

#### `services/fileReader.ts`

- サーバ上のファイルパスから本文を読む
- テキスト系はそのまま読む
- PDF や Office は当面サポート対象を限定する

#### `services/classifier.ts`

- OpenAI SDK を呼ぶ
- 固定カテゴリに分類する
- JSON 形式の結果を返す

#### `services/routing.ts`

- 読み込み結果と分類結果をまとめる
- Prisma で保存する
- API 応答用の結果を組み立てる

### API 例

#### 分類実行

`POST /api/routing/classify`

```json
{
	"filePath": "C:/data/invoice-001.txt",
	"originalName": "invoice-001.txt",
	"mimeType": "text/plain"
}
```

返却例:

```json
{
	"requestId": "route-xxx",
	"status": "classified",
	"category": "invoice",
	"confidence": 0.92,
	"reason": "請求書番号と金額が含まれるため"
}
```

#### 結果参照

`GET /api/routing/:requestId`

### Prisma スキーマ案

```prisma
model RoutingRequest {
	id           String   @id
	filePath     String
	originalName String
	mimeType     String
	status       String   @default("pending")
	category     String?
	confidence   Float?
	reason       String?
	model        String?
	promptVersion String?
	errorMessage String?
	createdAt    DateTime @default(now())
	updatedAt    DateTime @updatedAt

	@@map("routing_requests")
}
```

### 処理フロー

1. リクエストを受ける
2. `filePath` の存在確認を行う
3. 本文を読み込む
4. 必要なら簡易分割する
5. OpenAI SDK に渡す
6. 分類結果を Prisma へ保存する
7. 結果を HTTP レスポンスで返す

### 依存関係案

- `express`
- `@prisma/client`
- `prisma`
- `openai`
- `zod` または `yup`
- `typescript`
- `ts-node-dev`

### 実装上のポイント

- ファイルの中身はサーバ側のパスから読む前提にする
- OpenAI へ送る本文は必要最小限にする
- 同期処理なので、本文は小さめのファイルに限定する
- タイムアウト対策として、最大ファイルサイズを制限する
- 分類ルールはコード内ではなく設定値で持つ

### 最初に作る範囲

- `txt` / `md` / `csv` のみ対応
- 分類カテゴリは 3〜5 個
- 1 リクエスト = 1 ファイル
- OCR はこの段階では入れない

### この案の位置づけ

案1は、最小構成で動作確認したいときに向いている。まずは Prisma に保存して結果を返すところまで作り、その後にチャンク分割や OCR を足していく。

### 案2: 非同期ジョブキュー方式

1. アップロード時にファイルを保存し、ジョブを作成する
2. API は `jobId` だけ返す
3. ワーカーがキューからジョブを取得する
4. ファイル内容を抽出して OpenAI SDK で分類する
5. DB に分類結果と根拠を保存する
6. クライアントは結果取得 API でステータスを確認する

#### 向いているケース

- 遅いファイル抽出や大きめの文書を扱う
- 再試行やリトライを制御したい
- 実運用を見据えて安定性を重視したい

#### 利点

- API の応答を速く保てる
- OpenAI 失敗時に再実行しやすい
- 将来、複数ワーカーへ水平分割しやすい

### 案3: 同期API + 非同期フォールバックのハイブリッド

1. 小さいファイルは同期で分類する
2. 大きいファイルや抽出に時間がかかるものはジョブ化する
3. どちらも同じ分類ロジックを呼ぶ

#### 向いているケース

- 初期はシンプルにしたい
- ただし運用では重いファイルにも対応したい

#### 利点

- 最初の実装コストが低い
- 将来の拡張に繋げやすい

## 推奨案

本番を見据えるなら **案2の非同期ジョブキュー方式** を推奨する。

理由は、ファイル内容の抽出と LLM 推論は処理時間がぶれやすく、HTTP の同期処理に載せると不安定になりやすいため。

PoC なら案1でもよいが、後から案2へ寄せる前提で設計するのがよい。

## サービス構成案

### RoutingFile サービスの責務

- ファイル受付
- メタデータ保存
- 分類ジョブ作成
- ステータス確認 API
- 分類結果の保管

### Worker の責務

- ファイル本文の抽出
- OpenAI SDK 呼び出し
- 分類ラベルの確定
- 根拠やスコアの保存
- 失敗時のリトライ

### DB に持たせたい項目

- `fileId`
- `originalName`
- `mimeType`
- `size`
- `status` (`pending`, `processing`, `classified`, `failed`)
- `category`
- `confidence`
- `reason`
- `model`
- `promptVersion`
- `errorMessage`

## OpenAI SDK の使い方の方針

OpenAI SDK には、分類用のプロンプトを固定化して渡す。

入力には次を含める。

- ファイル名
- MIME タイプ
- 抽出した本文の先頭部分または必要範囲
- 必要ならメタデータ

出力は JSON 形式に寄せる。

例:

```json
{
	"category": "invoice",
	"confidence": 0.92,
	"reason": "請求書番号、金額、支払期限が含まれるため"
}
```

## 実装手順案

1. ファイル受付 API を作る
2. ファイル保存先を決める
3. 本文抽出モジュールを切り出す
4. OpenAI 呼び出しモジュールを切り出す
5. 分類ルールを固定のカテゴリ集合で定義する
6. 結果保存用の DB スキーマを追加する
7. ジョブキューまたはワーカーを追加する
8. ステータス照会 API を追加する

## エラー設計

- 抽出失敗は `extract_failed`
- OpenAI 失敗は `classification_failed`
- ファイル不正は `invalid_file`
- 再試行上限超過は `retry_exceeded`

## セキュリティと運用

- API キーは環境変数で管理する
- アップロードサイズ上限を設ける
- 送る本文は必要最小限にする
- 個人情報を含む場合はマスキングを検討する
- プロンプトとモデル名はバージョン管理する

## 推奨する最初の実装範囲

最初は次の小さな範囲で始めるのがよい。

- 対象ファイルは `txt` / `md` / `csv`
- 分類カテゴリは 3〜5 個程度
- 実装は非同期ジョブではなくてもよいが、API と分類処理は分離する
- 結果を DB に残す

## 結論

OpenAI SDK を使ったファイル分類は、単純な同期 API で始められるが、実運用を考えると RoutingFile サービスと分類ワーカーを分けた非同期構成が適している。

最初の一歩としては、ファイル受け取り、本文抽出、OpenAI 分類、結果保存、ステータス照会の5点を最小セットで作るのがよい。
