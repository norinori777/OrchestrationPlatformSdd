## 状態管理の設計方針

**`platform_requests` が正源、`saas_requests` はクライアント向けビュー**

```
ブラウザ          SaaS Backend          Platform          DB
  │                    │                    │               │
  │ POST /api/files     │                    │               │
  │───────────────────▶│                    │               │
  │                    │ INSERT pending ─────────────────────▶ saas.saas_requests
  │                    │ publish NATS event │               │
  │ 202 {requestId}    │                    │               │
  │◀───────────────────│                    │               │
  │                    │                    │               │
  │                    │     NATS event ───▶│               │
  │                    │                    │ UPSERT ────────▶ platform.platform_requests
  │                    │                    │               │  (pending → completed)
  │                    │ PATCH /api/requests/:id/status     │
  │                    │◀───────────────────│               │
  │                    │ UPDATE status ───────────────────────▶ saas.saas_requests
  │                    │                    │               │  (pending → completed)
  │ GET /api/files/:id │                    │               │
  │───────────────────▶│                    │               │
  │                    │ SELECT ─────────────────────────────▶ saas.saas_requests
  │ {status:completed} │                    │               │
  │◀───────────────────│                    │               │
```

Compacted conversationSaga の実装箇所をコードベースで特定します。まずロールバックや補償処理の定義があるかを近い範囲で確認します。


ロールバック付きの Saga パターンとしては、現状まだ実装されていません。

Saga のオーケストレーション本体は platformWorkflow.ts です。ここで pending → policy check → quota check → business process → completed という順方向の処理を管理しています。ただし、失敗時に過去の副作用を打ち消す compensate 処理は入っていません。

今あるのは次の 2 つです。

1. 状態を失敗系に更新する処理
platformWorkflow.ts と platformWorkflow.ts で、拒否・クォータ超過・キャンセル時に persistRequestActivity を呼んで denied / failed を記録しています。

1. 状態保存と SaaS 側同期
persistenceActivity.ts で platform_requests への upsert と SaaS Backend への PATCH コールバックをしています。これは状態整合のための処理で、業務データの取り消しではありません。

一方で、実際の業務副作用は index.ts の processRequestActivity から FileStorageService / UserService に対して POST / DELETE を送っていますが、それに対応する rollback 用の compensateActivity は定義されていません。つまり、今の構成は厳密には「Saga の補償トランザクション」ではなく、「Temporal による順方向オーケストレーション + 失敗時の状態更新」です。

必要なら次に、platformWorkflow.ts に補償アクティビティを追加して、create 後失敗時の delete や delete 前の退避復元まで含めた本当の Saga にできます。
